# inspiration-research - 灵感深度研究技能

## 触发条件

**语义匹配**（非精确关键词）：
- "记录灵感"、"灵感"、"深度研究"
- "帮我研究一下"、"分析一下这个想法"
- 直接发送思考内容（包含"？"、"是不是"、"应该...还是..."等探索性问题）

**不触发场景**：
- 简单的待办事项记录
- 会议纪要整理
- 事实性查询（如"今天天气"）

---

## 工作流程

### 1️⃣ 接收灵感

**输入**：用户的灵感文本（飞书消息、语音转文字等）

**处理**：
- 提取核心问题
- 识别待探索方向
- 标注标签（#AI 战略 #企业竞争力 等）

---

### 2️⃣ 拆解问题

**方法**：
- 将灵感拆解为 2-5 个具体问题
- 区分事实性问题 vs 观点性问题
- 标注优先级（核心问题 vs 衍生问题）

**输出格式**：
```markdown
## 💡 核心问题拆解

### 问题 1：[问题描述]
### 问题 2：[问题描述]
```

---

### 3️⃣ 搜索验证

**搜索策略**：
- 每个核心问题对应 1-2 个搜索词
- 中英文混合搜索（覆盖国内外观点）
- 优先选择 2025-2026 年的最新资料
- 目标：5-8 篇高质量参考资料

**搜索工具**：
- `web_search`（Brave Search API）
- 每次搜索间隔 2 秒（避免 API 限流）

---

### 4️⃣ 建立映射

**每篇资料的关联分析**：

| 关联类型 | 说明 | 示例 |
|---------|------|------|
| ✅ **验证** | 资料支持用户的观点 | "验证：你提到的 X 正在发生..." |
| ❌ **驳斥** | 资料反驳用户的观点 | "驳斥：你的假设 X 可能不成立，因为..." |
| 💡 **部分回答** | 资料回答问题的一部分 | "部分答案：X 是正确的，但 Y 需要补充..." |
| 🔍 **新视角** | 资料提供新的思考角度 | "新视角：从 Z 角度看，问题应该是..." |

**输出格式**：
```markdown
### 资料 N：[标题](URL)
**来源**: [来源] | **时间**: [时间]

**摘要**:
- [要点 1]
- [要点 2]

**💡 与你的思考关联**:
> **验证/驳斥/部分回答**：[具体关联内容]
```

---

### 5️⃣ 生成建议

**针对每个问题**：
- 资料的回答是什么
- 我的分析和建议
- 待验证的假设

**输出格式**：
```markdown
### 针对你的问题 N：[问题描述]

**你的假设**：[用户的观点]

**资料的验证**：
| 你的假设 | 资料验证 | 结论 |
|---------|---------|------|
| ... | ... | ... |

**我的建议**：[具体建议]
```

---

### 6️⃣ 输出报告

**报告版本选项**：

| 版本 | 适用场景 | 内容结构 |
|------|----------|----------|
| **完整版**（默认） | 深度研究、存档 | 原始灵感 + 问题拆解 + 5-8 篇资料 + 核心洞察 + 行动建议 |

## 👤 用户风格偏好（2026-02-27 确认）

### 汇报文章风格

**语言风格**：
- ✅ 简洁直接：去掉"这句话让我想了很多"等过渡语
- ✅ 数据精炼：只保留最核心的数据（如 24%、17%、321 万片）
- ✅ 保留类比：有力的类比保留（如"电力普及之后"）
- ❌ 不过度口语：去掉"很有意思的是"、"带着这个问题"等表达
- ❌ 不强结构：不用"一二三四五六"编号，用分隔线自然过渡
- ❌ 不要建议：汇报场景不生成行动建议

**结构要求**：
- 标题：问题式，简洁直接
- 开篇：直接点明灵感来源，快速引出核心问题（1-2 段）
- 小标题：简短有力，直接表达核心观点
- 段落：每段 2-3 句话（数据 + 观点 + 结论）
- 列表：每项一行，简洁呈现

**示例**（用户确认的最终版本）：
```
算力平权时代，什么才是企业的核心竞争力？
本周钉钉团队沟通时，他们提到一个判断："算力将会最终平权"...

算力正在变成基础设施
算力平权已经在发生，2026 年全球芯片制造企业 12 英寸产能建设设备支出增长 24%...
算力本身，正在变成基础能力，而非差异化优势。

数据成为差异化因素
IDC 调研显示，企业 AI 能力主要受业务数据约束...
就像电力普及之后，核心竞争力不是谁能拿到电，而是谁能用电创造出更好的产品。
```

## 📝 飞书文档格式规范（2026-02-27 更新）

### 格式要求（强制）

**目标**：让飞书自动解析 Markdown 为丰富的块格式（Heading、Bullet、Divider 等）

**标题层级**：
- `# 标题` → H1（主标题，全文 1 个）
- `## 标题` → H2（大章节）
- `### 标题` → H3（子章节/资料标题）

**列表格式**：
- `- 项目` 或 `* 项目` → Bullet（无序列表）
- `1. 项目` → Ordered（有序列表）
- `- [ ] 任务` → Todo（待办事项）

**文本样式**：
- `**加粗**` → **Bold**（用于强调关键词、标签、假设编号）
- `*斜体*` → *Italic*
- `[链接文本](URL)` → 可点击链接

**分割线**：
- `---` → Divider（分隔大章节）

### 飞书写入技巧（2026-02-28 更新）

**使用 `feishu_doc` 工具**：

```python
# 方式 1：write（覆盖写入，适合首段或空文档）
feishu_doc(action="write", doc_token="xxx", content="Markdown 内容")

# 方式 2：append（追加内容，适合分段写入）
feishu_doc(action="append", doc_token="xxx", content="追加内容")
```

**核心原则**：
- ✅ **分段写入**：每次控制在 1500-2000 字以内（避免 400 错误）
- ✅ **先 write 后 append**：首段用 write，后续用 append
- ✅ **让飞书自动解析 Markdown**：不要使用块级 API
- ❌ **避免块级 API**：block_type 7 文本块会失败（invalid param）
- ❌ **避免单次过长**：长文档必须分多次写入

| 第 3 段 | 第 4-5 条灵感 | ~1000 字 | append |
